================================================================================
GUÍA DOCENTE: SISTEMA DE INVENTARIO PARA RESTAURANTE
================================================================================

Esta guía está pensada para que la leas en voz alta mientras compartes pantalla
con tu audiencia. Puedes mostrar los archivos con `cat`, `ls`, `tree`, etc., y
ejecutar los comandos uno por uno mientras explicas qué significa cada uno.

Objetivo de la sesión:
    - Entender cómo un frontend, un backend y una base de datos colaboran.
    - Ver que Python es el "cerebro" que toma decisiones y organiza datos.
    - Entender por qué separamos el código en capas (layers).
    - Saber ejecutar y verificar el proyecto paso a paso.

================================================================================
1. INTRODUCCIÓN: ¿QUÉ ES ESTE PROYECTO?
================================================================================

"Este es un sistema de inventario para un restaurante. Su trabajo es recordar
qué ingredientes tenemos, cuánto stock queda, quién nos los vende y cada vez
que entra o sale mercancía.

Lo hemos construido en capas para que sea fácil de entender, probar y
escalar. Vamos a ver cada capa como si fueran diferentes personajes en una
cocina."

Metáfora visual:
    - El CLIENTE (navegador / terminal) es el comensal que pide un plato.
    - El CAMARERO (frontend / CLI) toma la orden y la lleva a la cocina.
    - El CHEF (backend / Python) recibe la orden, decide qué hacer y usa los
      ingredientes según las reglas del restaurante.
    - LA NEVERA (base de datos / SQL) es donde se guarda todo.
    - EL LIBRO DE REGISTRO (modelos / SQLAlchemy) describe dónde va cada cosa
      en la nevera.

3. DIFERENCIA ENTRE FRONTEND, BACKEND Y BASE DE DATOS
================================================================================

FRONTEND:
    "Es todo lo que el usuario ve y toca. Botones, colores, formularios,
     tablas. En este proyecto está en web/static/index.html y web/static/app.js.
     El frontend no guarda nada por sí mismo; solo pide datos al backend y los
     muestra bonitos."

    Metáfora: El menu del restaurante y el camarero. Tú pides por el menu,
    pero el menu no cocina.

BACKEND:
    "Es el cerebro. Es código Python que recibe peticiones, aplica reglas de
     negocio y decide qué hacer. Por ejemplo: 'No puedes usar más stock del
     que tienes' o 'Cada movimiento debe quedar registrado'.

     En este proyecto el backend principal está en src/inventory/inventory.py.
     La API web que habla con el frontend está en web/api.py."

    Metáfora: El chef. Toma la orden, cocina según la receta y devuelve el
    plato.

BASE DE DATOS:
    "Es donde se guardan los datos de forma permanente. Si apagamos el
     servidor, la base de datos sigue teniendo la información.

     Usamos SQLite para desarrollo local (un archivo .db) y PostgreSQL para
     producción/Docker. La base de datos no decide nada; solo almacena."

    Metáfora: La nevera y el almacén. Guarda ingredientes y el historial de
    qué se ha usado.

================================================================================
4. CAPAS DEL CÓDIGO PYTHON
================================================================================

Muestra cada archivo y explica su papel.

    cat src/inventory/config.py

"Config.py es como el cuaderno de configuración del restaurante. Aquí decidimos
si usamos SQLite o PostgreSQL, el nivel de logs, etc. Es importante porque en
desarrollo usamos SQLite y en producción PostgreSQL, pero el resto del código
no necesita saberlo."

    cat src/inventory/database.py

"Database.py es la puerta de la nevera. Crea el motor de SQLAlchemy y nos da
sesiones. Una sesión es como una bandeja temporal: cargamos ingredientes, los
modificamos, y luego decimos 'guarda los cambios' (commit) o 'descarta todo'
(rollback).

El context manager `with self.db.session()` garantiza que si algo falla, se
revertan los cambios y se cierre la conexión."

    cat src/inventory/models.py

"Models.py es el plano de la nevera. Define qué tablas existen y cómo se
relacionan.

Por ejemplo, Ingredient tiene:
    - sku: código único del producto.
    - current_stock: cuánto tenemos ahora.
    - reorder_level: abajo de qué cantidad debemos pedir más.
    - category_id: a qué categoría pertenece.

La línea `relationship` dice: 'un ingrediente pertenece a una categoría'. Eso
permite hacer ingredient.category.name sin escribir SQL manual."

    cat src/inventory/inventory.py

"Inventory.py es el chef. Aquí viven las reglas del negocio.

Por ejemplo, receive_stock hace esto:
    1. Convierte la cantidad a Decimal (para no perder precisión con decimales).
    2. Abre una sesión con la base de datos.
    3. Busca el ingrediente.
    4. Suma la cantidad al stock actual.
    5. Crea una fila en stock_transactions (pista de auditoría).
    6. Guarda todo.

Si intentas usar más stock del que hay, el chef se niega y lanza
InsufficientStockError."

    cat src/inventory/cli.py

"Cli.py es el camarero de la terminal. No decide nada; solo convierte lo que
el usuario escribe en llamadas al chef (InventoryManager). Luego imprime el
resultado en pantalla."

5. ¿QUÉ PASA CUANDO USAS LA UI WEB?
================================================================================

Paso a paso, mientras haces clic en "Receive" en el navegador:

1. El usuario hace clic en "Receive", rellena el formulario y pulsa Save.
2. app.js construye un objeto JSON: { "quantity": 10, "reference": "PO-123" }.
3. app.js usa fetch() para enviar una petición HTTP POST a
   /api/ingredients/1/receive.
4. El navegador envía la petición por la red.
5. FastAPI (web/api.py) recibe la petición, valida el JSON con Pydantic y llama
   a InventoryManager.receive_stock(1, 10).
6. InventoryManager abre una sesión de base de datos, actualiza el stock,
   crea la transacción y guarda.
7. El ingrediente actualizado se convierte a JSON y se devuelve al navegador.
8. app.js recibe la respuesta, actualiza la tabla y muestra un mensaje verde.

Todo esto ocurre sin recargar la página.

6. ¿QUÉ ES UNA PETICIÓN HTTP?
================================================================================

"HTTP es el idioma que usan el navegador y el servidor para hablar.

Una petición tiene:
    - Método: GET (pedir datos), POST (crear algo), PUT/PATCH (actualizar),
      DELETE (borrar).
    - URL: la dirección, por ejemplo /api/ingredients.
    - Headers: metadatos, como Content-Type: application/json.
    - Body: el contenido, por ejemplo { "quantity": 10 }.

La respuesta tiene:
    - Código de estado: 200 OK, 201 Created, 404 Not Found, 422 Unprocessable.
    - Body: los datos en JSON."

Demuestra con curl:

    curl -X GET http://localhost:8000/api/ingredients
    curl -X POST http://localhost:8000/api/categories \
         -H "Content-Type: application/json" \
         -d '{"name":"Produce"}'

================================================================================
7. EL CEREBRO PYTHON: ¿CÓMO ORDENA Y DECIDE?
================================================================================

"Python no es magia. Cuando llega una petición, sigue instrucciones claras.

Por ejemplo, cuando recibimos stock:
    1. Valida: convierte el texto '10' en Decimal('10').
    2. Busca: pregunta a la base de datos por el ingrediente con id=1.
    3. Decide: si no existe, lanza NotFoundError.
    4. Calcula: ingredient.current_stock += quantity.
    5. Registra: crea un StockTransaction para dejar evidencia.
    6. Persiste: commit guarda los cambios; rollback los deshace si hay error.
    7. Responde: devuelve el ingrediente actualizado.

Esto se llama lógica de negocio. Es el conjunto de reglas que hacen que el
sistema sea útil y confiable."

8. PERMISOS Y BASE DE DATOS
================================================================================

"La base de datos tiene usuarios y permisos. En Docker usamos un usuario
llamado 'inventory' con contraseña 'inventory'. En producción nunca usarías
una contraseña tan simple.

SQLite es un archivo común (.db). Cualquiera que pueda leer el archivo puede
leer la base de datos. Por eso en producción preferimos PostgreSQL, donde el
servidor controla quién puede conectarse y qué puede hacer."

Mostrar cómo entrar a PostgreSQL con Docker:

    docker compose exec db psql -U inventory -d inventory

Dentro de psql:

    \dt                    -- lista tablas
    SELECT * FROM ingredients;
    SELECT * FROM stock_transactions;
    \q                     -- salir

Para SQLite:

    sqlite3 data/inventory.db
    .tables
    SELECT * FROM ingredients;
    .quit

10. COMANDOS PARA INSTALAR Y CONFIGURAR
================================================================================

Crear entorno virtual e instalar dependencias:

    cd restaurant-inventory
    python -m venv .venv
    source .venv/bin/activate
    pip install -r requirements.txt
    pip install -e ".[dev,web]"

¿Qué significa cada uno?
    - python -m venv .venv: crea un entorno virtual aislado.
    - source .venv/bin/activate: activa el entorno.
    - pip install -r requirements.txt: instala dependencias principales
      (SQLAlchemy, psycopg2, python-dotenv).
    - pip install -e ".[dev,web]": instala el proyecto en modo editable
      más dependencias de desarrollo (pytest, ruff, mypy) y web (FastAPI).

11. COMANDOS PARA CORRER LA APLICACIÓN
================================================================================

Inicializar la base de datos CLI:

    python -m inventory init

Ejemplo de uso del CLI:

    python -m inventory add-category Produce
    python -m inventory add-ingredient TOM-001 "Roma Tomatoes" kg --current-stock 12.5
    python -m inventory list-ingredients
    python -m inventory receive 1 10

Correr el servidor web:

    cd web
    export PYTHONPATH=../src
    uvicorn api:app --reload

Abrir en el navegador:
    http://localhost:8000           -> Interfaz web.
    http://localhost:8000/docs      -> Documentación interactiva de la API.

Correr con Docker:

    docker compose up --build

================================================================================
12. COMANDOS DE CALIDAD DE CÓDIGO
================================================================================

RUFF: linter y formateador.
    "Ruff revisa el código buscando errores de estilo, imports no usados,
     nombres de variables incorrectos, etc. También puede formatear el código."

    ruff check src tests web
    ruff check --fix src tests web
    ruff format src tests web
    ruff format --check src tests web

MYPY: verificador de tipos.
    "Python es de tipado dinámico, pero nosotros añadimos anotaciones de tipo.
     mypy las revisa para detectar errores antes de ejecutar el código."

    mypy src/inventory web

PYTEST: ejecutor de pruebas.
    "pytest corre los tests y verifica que el sistema sigue funcionando. Si
     cambias algo y rompes una regla de negocio, pytest lo detecta."

    pytest
    pytest -q              -> modo silencioso.
    pytest --cov=inventory -> con cobertura.

Combinación completa de verificación:

    ruff check src tests web && ruff format --check src tests web && mypy src/inventory web && pytest -q
